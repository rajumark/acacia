# 🚀 AI-Native Compose DSL (Shortify)

> A Gradle plugin that introduces an **AI-optimized DSL layer** over Jetpack Compose by compressing verbose APIs into a deterministic, token-efficient syntax.

---

## 📌 Vision

Modern development is shifting from:

```
Human → writes code → IDE assists
```

to:

```
Human intent ↔ AI ↔ code ↔ compiler
```

This project builds the missing layer:

> **A deterministic, AI-friendly abstraction over Compose UI**

---

## 🎯 Goals

* Reduce verbosity of Compose APIs
* Create **predictable patterns for AI generation**
* Improve **token efficiency for LLMs**
* Enable **bidirectional transformation**
* Provide **zero runtime overhead (compile-time only)**

---

## 🧠 Core Concept

We introduce a **short DSL layer** over Compose.

### Example

#### Input (DSL)

```kotlin
Modifier.p(16).bg(Red).fmw()
```

#### Output (Generated Compose)

```kotlin
Modifier
    .padding(16.dp)
    .background(Color.Red)
    .fillMaxWidth()
```

## ⚙️ How It Works

### Step 1: Plugin Applied

```kotlin
plugins {
    id("com.shortify")
}
```

---

### Step 2: Gradle Task Runs

```
generateShortModifiers
```

---

### Step 3: Dependency Scan

* Reads `compileClasspath`
* Extracts `.jar` and `.aar`
* Filters Compose libraries

---

### Step 4: API Extraction

* Parses Modifier extension functions
* Builds internal model

---

### Step 5: Code Generation

Creates:

```
build/generated/source/shortify/ShortModifiers.kt
```

---

### Step 6: Compilation

Generated code becomes available automatically.

# 🧱 1. NEVER Hardcode Paths

### ❌ Bad

```kotlin
val file = File("/Users/dev/project/libs/compose.jar")
```

### ✅ Good

```kotlin
val files = project.configurations
    .getByName("compileClasspath")
    .resolve()
```

### 💡 Why

* Works across machines, CI, OS
* Gradle manages dependency resolution

---

# ⚙️ 2. Always Use Gradle APIs (Not DIY)

### ❌ Bad

```kotlin
File("build/generated/source")
```

### ✅ Good

```kotlin
val outputDir = project.layout.buildDirectory
    .dir("generated/source/shortify")
    .get()
    .asFile
```

### 💡 Why

* Supports configuration cache
* Works with Gradle internals safely

---

# 🧩 3. Use `tasks.register`, NOT `tasks.create`

### ❌ Bad

```kotlin
project.tasks.create("generateDsl")
```

### ✅ Good

```kotlin
val task = project.tasks.register("generateDsl", GenerateDslTask::class.java)
```

### 💡 Why

* Lazy configuration
* Faster builds

---

# 🔁 4. Avoid `afterEvaluate` When Possible

### ❌ Bad

```kotlin
project.afterEvaluate {
    // logic
}
```

### ✅ Good

```kotlin
project.tasks.named("preBuild") {
    dependsOn(task)
}
```

### 💡 Why

* `afterEvaluate` breaks configuration cache
* Hard to debug lifecycle issues

---

# 📦 5. Use Artifact Metadata (NOT filename)

### ❌ Bad

```kotlin
if (file.name.contains("compose"))
```

### ✅ Good

```kotlin
val artifacts = configuration
    .resolvedConfiguration
    .resolvedArtifacts

val compose = artifacts.filter {
    it.moduleVersion.id.group.startsWith("androidx.compose")
}
```

### 💡 Why

* Filenames can change
* Metadata is reliable

---

# 📁 6. Use `Provider` API (Lazy Values)

### ❌ Bad

```kotlin
val dir = project.buildDir
```

### ✅ Good

```kotlin
val dir = project.layout.buildDirectory.dir("shortify")
```

### 💡 Why

* Enables lazy evaluation
* Required for modern Gradle

---

# ⚡ 7. Declare Inputs & Outputs (CRITICAL)

### ❌ Bad

```kotlin

```

### ✅ Good

```kotlin
@InputFiles
val classpath = project.files()

@OutputDirectory

```

### 💡 Why

* Enables incremental builds
* Enables caching
* Huge performance gain

# 🧠 8. Make Task Incremental-Friendly

### 💡 Why

Gradle can skip task if inputs unchanged.

# 🧾 9. Use KotlinPoet (Never String Concatenation)

### 💡 Why

* Handles imports automatically
